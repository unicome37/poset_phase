# Contributing Guide

欢迎参与 `poset_phase`。本指南给出最小协作规则，确保提交可审阅、可复现、可回滚。

## 1. 分支与提交流程

- 默认目标分支：`main`
- 每次提交保持**单一目的**，避免把配置、结果、论文内容混在同一个 commit
- 提交信息建议采用 `<type>(<scope>): <summary>`

详细模板与分批策略见：

- [`COMMIT_STRATEGY.md`](./COMMIT_STRATEGY.md)

## 2. 原子提交分类（强烈建议）

按以下顺序拆分：

1. `config`（实验配置）
2. `outputs`（结果与审计产物）
3. `paper`（论文与投稿资产）

如必须跨类改动，请拆成多个提交并标明依赖关系。

## 3. 行尾与文件规范

仓库已通过 `.gitattributes` 固化换行规则：

- 文本文件统一使用 LF（仓库内）
- Windows 脚本（`.bat/.cmd/.ps1`）使用 CRLF
- 图像/压缩包/PDF 等按二进制处理

请不要手动批量改动无关文件的行尾，避免噪音 diff。

## 4. 结果文件与可复现性

- 结果文件命名需包含实验 scope（必要时加日期）
- `outputs*` 下新增结果应附最小解释（参数、seed、关键结论）
- 提交前确认未混入敏感信息（token/key/credential）

## 5. PR/评审建议

- 一次 PR 只解决一个问题域
- 描述应包含：动机、变更范围、验证方式、回滚方案
- 若含图文更新，请标注对应章节/图号变化

---

若你不确定某文件属于哪一类提交，先按 `COMMIT_STRATEGY.md` 分类，再执行暂存与提交。
